Ch 1: Principles of Software Architecture

Core Question

What separates design from architecture?

Architecture = decisions that are hard to reverse and have systemic impact.

SOLID + DRY + YAGNI

  • SRP — one reason to change
  • OCP — open to extension, closed to modification
  • DIP — depend on abstractions, not concretions
  • DRY — one authoritative source for every fact
  • YAGNI — don't build what you don't need today

Design vs Architecture — examples

  • "Constructor or factory function?" → design, easy to change
  • "Monolith or microservices?" → architecture, very hard to change
  • "Which library for JSON?" → design
  • "Synchronous or event-driven?" → architecture

Event-Driven Architecture in C++

Key Benefit

Zero source-code dependencies between subsystems. Adding a new subscriber requires zero changes to the publisher.

Trade-offs

  • ✅ Loose coupling — subsystems evolve independently
  • ✅ Easy to add new consumers
  • ⚠️ Harder to trace execution flow
  • ⚠️ Ordering and deduplication are your problem
template<typename Event>
class event_bus {
  std::vector<std::function<void(const Event&)>> h_;
public:
  void subscribe(std::function<void(const Event&)> h)
    { h_.push_back(std::move(h)); }
  void publish(const Event& e)
    { for (auto& h : h_) h(e); }
};

// Typed: can't publish to wrong topic
event_bus<order_placed> order_bus;
event_bus<payment_processed> pay_bus;

Ch 4: System Design Challenges

Eight Fallacies of Distributed Computing (Deutsch, 1994)

  1. The network is reliable
  2. Latency is zero
  3. Bandwidth is infinite
  4. The network is secure
  5. Topology doesn't change
  6. There is one administrator
  7. Transport cost is zero
  8. The network is homogeneous

Most Dangerous in Practice

Fallacy 1: The network is reliable.

Engineers build a service calling 5 others synchronously. In production, one has a brief hiccup → cascade failure within minutes.

Fix: Design for Failure

  • Circuit breakers on all downstream calls
  • Timeouts everywhere
  • Retries with exponential backoff
  • Graceful degradation
Part II — Design & Development11 / 38

CAP Theorem

A distributed system can guarantee at most two of three:

  • Consistency — every read sees the latest write
  • Availability — every request gets a response
  • Partition Tolerance — works despite network splits

Key Insight

Partition tolerance is not optional in a real distributed system — networks partition. The real choice is C vs A when a partition occurs.

ChoiceBehaviourExamples
CAImpossible in practice—
CPConsistent; may reject writes on partitionHBase, ZooKeeper
APAvailable; may return stale dataDynamoDB, Cassandra

In Practice

Financial transactions → CP. User preference cache → AP. Different parts of your system can make different choices.

CQRS Pattern in C++

Command Query Responsibility Segregation

Separate the write model (commands) from the read model (queries). Optimise each side independently.

Benefits

  • Write side: validates invariants, appends events
  • Read side: denormalised projections, fast queries
  • Independent scaling of reads vs writes
  • No N+1 query problem on read side
// Write side — validates + produces events

  std::string customer_;
  double amount_;
public:
  void validate() const; // throws on bad data
};

// Read side — optimised projection

public:
  std::vector<order_dto> execute();

Strong Types

// ❌ Weak types: silent bug — args in wrong order
void create_account(int user_id, int account_id);
create_account(42, 17);  // is 42 user or account?

// ✅ Strong types: compiler catches transposition
struct user_id    { int value; };
struct account_id { int value; };
void create_account(user_id, account_id); // safe

// [[nodiscard]] forces caller to handle result
[[nodiscard]]
std::optional<order_id> place_order(...);

Zero Runtime Overhead

A trivial struct wrapper with no virtual methods compiles to the same code as a plain int. The protection is purely compile-time.

Where to Use

  • IDs of different entity types
  • Measurements with units (meters vs feet)
  • Flags and status values
  • Indices into different containers

Part II — Design & Development16 / 38

CRTP: Static Polymorphism

// CRTP — compile-time polymorphism, zero vtable
template<typename Derived>
struct serialisable {
  std::string serialise() const {
    return static_cast<const Derived*>(this)
               ->do_serialise();
    // resolved at compile time — inlined
  }
};

struct order : serialisable<order> {
  std::string do_serialise() const
    { return "Order{...}"; }
};

order o;
o.serialise(); // no virtual call

Zero Virtual Dispatch

The compiler resolves the dispatch statically. The call is inlined. Used in the STL: std::char_traits, iterator adapters.

When to Use CRTP

  • Tight inner loops (physics, rendering)
  • Real-time / embedded systems
  • Policy mix-ins with multiple base classes
  • When vtable cost is measurable

# Conan 2 — dependency management
# conan install . --build=missing
# cmake --preset conan-release

Key CMake Idioms (≥ 3.16)

  • PRIVATE — internal use only
  • PUBLIC — propagates to linkers of this target
  • INTERFACE — propagates only, not used by this target

Never use global variables

Avoid include_directories(), link_libraries(). Use target-scoped versions — they prevent property leakage across targets.

Ch 8: Writing Testable Code

Testing Pyramid

        /   E2E   \       few, slow, brittle
       /  Integration \   more, medium speed
      /  Unit Tests    \  many, fast, isolated

Root Cause of Untestable Code

A class instantiates its dependencies with new inside the constructor. You cannot test it in isolation — not a testing problem, a design problem.

Fix: Constructor Injection

If a class receives its dependencies through the constructor, you can inject fakes in tests and real implementations in production.

Test Double Taxonomy

  • Stub — returns hardcoded values; no assertions
  • Fake — working lightweight implementation (in-memory DB)
  • Mock — verifies specific interactions were made
  • Spy — records calls; assertions happen after the act

Dependency Injection in C++

class order_service {
  i_payment_gateway& payment_;
  i_inventory&        inventory_;
public:
  order_service(i_payment_gateway& p,
                i_inventory& i)
    : payment_(p), inventory_(i) {}
  // service only knows about interfaces
};

// In tests: inject fakes
fake_payment    fp;
fake_inventory  fi;
order_service   svc(fp, fi);
// assert behaviour in isolation

// In production: inject real impls
stripe_gateway  pg;
db_inventory    di;
order_service   prod(pg, di);

No Framework Needed

Constructor injection is sufficient in C++. No Spring/Guice equivalent required. Keep it simple — the container is your main().

Rules

  • No new ConcreteImpl() inside a service
  • No Singleton::get_instance()
  • No global state
  • Depend on interfaces, not concretions

Ch 9: CI/CD

CI Pipeline Stages

git push → build → test → lint
  → package → staging deploy
  → integration test → production

Most Important Rule

Build once, deploy the same artefact everywhere.

Do not build in staging, build in prod, build in dev. Build once in CI, push to a registry, deploy that exact binary everywhere. Eliminates "it worked in staging" bugs.

Key Practices

  • Every commit builds and tests
  • Feature flags over long-lived branches
  • Automated rollback on health probe failure
  • Trunk-based development

C++ Specific

  • Cache build objects (ccache)
  • Run sanitisers in CI (not just locally)
  • Enforce clang-format in PR checks
  • Publish coverage reports

Ch 10: Security by Design

OWASP Top 10 — C++ Relevance

  • A01 Broken Access Control — check permissions at every layer
  • A02 Cryptographic Failures — use system TLS; never roll your own crypto
  • A03 Injection — validate and sanitise all external input
  • A05 Security Misconfiguration — no debug endpoints, no default passwords
  • A09 Logging Failures — log security events; do not log secrets

Secure Design Principles

  • Least privilege — request only needed permissions
  • Defence in depth — multiple independent layers
  • Fail secure — deny by default; whitelist
  • Zero trust — verify every request, every time

Threat Modelling

For each system entry point: who can reach it, what can they send, what can go wrong? Document and mitigate.

Secure C++ Idioms

// ❌ Classic buffer overflow pattern
void process(char* data, int len);

// ✅ std::span — bounds are part of the type
void process(std::span<const std::byte> data) {
  if (data.size() > MAX_PAYLOAD)
    throw std::invalid_argument{"too large"};
}

// ✅ Checked arithmetic
auto safe_add(int a, int b)
  -> std::optional<int> {
  if (b > 0 && a > INT_MAX - b)
    return std::nullopt; // overflow
  return a + b;
}

std::span is the Replacement

The classic void* data, size_t len pattern has caused countless CVEs. std::span bundles the pointer and size — you cannot accidentally pass a span of the wrong length.

Security Checklist

  • Validate at every external boundary
  • Use RAII for resource management
  • Enable stack protector + FORTIFY_SOURCE
  • Run ASan in CI to catch buffer issues early

Ch 11: Performance Engineering

Performance Workflow

Measure → Profile → Identify hotspot
  → Optimise → Measure again

Never optimise without measurement first.

Profiling Tools

  • perf (Linux) — CPU profiling, cache stats
  • Instruments (macOS) — time profiler, allocations
  • valgrind --tool=callgrind
  • Google Benchmark — micro-benchmarks
  • VTune, Perf Studio

Common Performance Anti-Patterns

  • Optimising without profiling (guessing)
  • Optimising 0.1% of runtime while ignoring 60%
  • Premature optimisation (YAGNI)
  • Ignoring allocator behaviour

Low-Hanging Fruit

  • Pass by const& for large objects
  • reserve() before bulk pushes
  • Move semantics for temporaries
  • Avoid virtual calls in tight loops

Cache-Friendly Design

// AoS — interleaves hot and cold data
struct particle {
  float x, y, z;         // hot (physics loop)
  float vx, vy, vz;      // hot (physics loop)
  float mass, health;    // cold — wastes cache!
};
// updating positions loads mass+health needlessly

// SoA — hot data together in memory
struct particles {
  std::vector<float> x, y, z;     // hot
  std::vector<float> vx, vy, vz;  // hot
  std::vector<float> mass, health; // cold
};
// position loop only touches x/y/z/vx/vy/vz

Why It Matters

  • Cache line = 64 bytes
  • L1 cache latency: ~1 ns
  • Main memory latency: ~100 ns
  • SoA can be 2–4× faster in tight loops

Other Cache Optimisations

  • Row-major traversal (C++ arrays)
  • Prefetch with __builtin_prefetch
  • Avoid pointer-chasing (linked lists)
  • Align hot data to cache lines

Ch 13: Designing Microservices

Decomposition by Bounded Context (DDD)

  • Identify bounded contexts → one service per context
  • Each service owns its data — no shared databases
  • Communicate via events (async) or APIs (sync)

Anti-Patterns

  • Distributed monolith — micro deployments, tight coupling
  • Shared DB — breaks independent deployment
  • Nanoservices — overhead > benefit

Service Decomposition Example

  • order-service — owns orders DB
  • payment-service — owns payment + ledger DB
  • inventory-service — owns stock DB
  • notification-service — owns templates DB

Communication

  • Sync (gRPC/HTTP): use when you need an immediate result
  • Async (events): use when temporal decoupling is acceptable

The Saga Pattern

// Distributed transaction without 2PC
OrderPlaced
  └→ PaymentService.ChargeCard
       ├ success → InventoryService.ReserveStock
       │              ├ success → Order.Confirm
       │              └ failure → PaymentService.RefundCard
       │                          → Order.Cancel
       └ failure → Order.Cancel

// Each step has a compensating transaction:
// ChargeCard ← compensated by RefundCard
// ReserveStock ← compensated by ReleaseStock

Why Sagas?

You cannot use database transactions across service boundaries. Sagas replace one atomic transaction with a sequence of local transactions, each with a compensating transaction that rolls back the effect on failure.

Critical Rule

Design the compensating transactions when you design the forward transactions — not as an afterthought. See ch13_designing_microservices/src/microservice_demo.cpp.

Observability

Logs — what happened

Structured JSON with correlation ID propagated through all downstream calls.

{"level":"info","trace_id":"abc",
 "service":"order","msg":"placed"}

Metrics — how the system performs

RED method: Rate · Errors · Duration (p50/p95/p99)

USE method (infra): Utilisation · Saturation · Errors

Traces — how a request flowed

Distributed tracing with span context propagation across all service boundaries. Standard: OpenTelemetry.

Observability is Not Optional

In a microservices system, a single user request might touch 10 services. Without distributed tracing, production debugging is impossible.

Correlation ID

Generated at the edge (API gateway). Every service propagates it in logs and in downstream call headers. Enables full trace reconstruction.

Ch 14: Containers

# Multi-stage Dockerfile for C++
FROM ubuntu:22.04 AS builder
RUN apt-get install -y cmake g++ libssl-dev
COPY . .
RUN cmake -B build -DCMAKE_BUILD_TYPE=Release \
  && cmake --build build -j$(nproc)

# Runtime: only binary + runtime deps (~150 MB)
FROM ubuntu:22.04 AS runtime
RUN apt-get install -y libssl3 \
  && rm -rf /var/lib/apt/lists/*
COPY --from=builder /build/my_service /app/
RUN useradd -r app && chown app:app /app/my_service
USER app
ENTRYPOINT ["/app/my_service"]

Why Multi-Stage?

Build stage: ~2 GB (compiler, CMake, headers). Runtime stage: ~150 MB. Smaller image = faster pull, smaller attack surface.

Never:

  • Run as root inside a container
  • Bake secrets into image layers
  • Use latest tag in production

Kubernetes Essentials

# Every service needs all three:
resources:
  requests: {cpu: "100m", memory: "128Mi"}  # scheduling
  limits:   {cpu: "500m", memory: "512Mi"}  # enforcement

readinessProbe:
  httpGet: {path: /health, port: 8080}
  initialDelaySeconds: 5    # load balancer gate

livenessProbe:
  httpGet: {path: /health, port: 8080}
  periodSeconds: 10         # restart if fails

Three Mandatory Resources

  • Resource requests + limits: without them, pods are scheduled on overloaded nodes and OOM-killed
  • readinessProbe: without it, traffic is routed before the app is ready
  • livenessProbe: without it, a deadlocked pod is never restarted

Never deploy without all three.

Twelve-Factor App — Critical Factors

FactorRule
III. ConfigEnv vars, never in code

XI. Logsstdout only; never write log files

Stateless Processes (Factor VI)

If your service stores state in memory — sessions, caches, in-flight data — you cannot scale horizontally or restart freely. Everything must live in external backing services.

Graceful Shutdown (Factor IX)

Handle SIGTERM: drain in-flight requests, close DB connections, exit cleanly. Kubernetes sends SIGTERM 30s before force-killing.

Cloud-Native Resilience Patterns

Circuit Breaker

Closed → failures exceed threshold → Open
Open → timeout elapsed → Half-Open
  → probe success → Closed
  → probe failure → Open

When Open: fail fast, return cached result or error — don't call downstream. Thread pool stays healthy. No cascade.

Bulkhead

Isolate resources per dependency. Separate thread pools for each downstream service — one slow DB cannot exhaust the pool for all services.

Without Circuit Breaker

Downstream slow → requests queue up → thread pool exhausts → your service slows → upstream queues → cascade failure.

Retry with Exponential Backoff + Jitter

wait = min(base × 2^attempt
           + jitter, max_wait)

Jitter prevents thundering herd on recovery.

Service Mesh

What a Mesh Provides (Transparent to App)

  • mTLS — all east-west traffic encrypted + authenticated
  • Retries — automatic with exponential backoff
  • Circuit breaking — at the proxy layer
  • Distributed tracing — auto-inject span headers
  • Traffic management — canary, A/B, blue-green

Trade-off

Application code becomes simpler. Operational complexity of the control plane increases. For large teams with many services: usually worth it.

Summary & Key Takeaways

1. Architecture is about trade-offs — there is no universally best style
2. Design for change — SOLID, DDD, interfaces keep systems maintainable
3. C++20 is expressive and safe — use optional, span, variant, concepts
4. Testability is designed in — dependency injection, interfaces, seams from day one

Practice: Write Architecture Decision Records

Take a real system you're working on. Write an ADR for three of its most significant structural decisions:

  • Context — what forced the decision?
  • Alternatives considered — what else was on the table?
  • Trade-offs — what does each option cost?
  • Decision — what did you choose and why?
  • Consequences — what does this constrain in future?

The act of writing forces clarity. Clarity exposes gaps in reasoning.
